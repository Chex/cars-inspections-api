# Cars and Inspections API (Framework Scaffold)

This is an assignment scaffold implemented in PHP with a Laravel-inspired layered structure.

## Included

- API entrypoint in `public/index.php`
- Four routes:
  - `GET /cars`
  - `POST /cars`
  - `GET /inspections`
  - `POST /inspections`
- Layered structure:
  - `src/Controllers`
  - `src/Services`
  - `src/Repositories`
  - `src/Validation`
  - `src/Support`
- File-based persistence in `storage/*.json`
- Basic validation and structured error responses
- Safe file writes with `LOCK_EX` and monotonic id assignment based on current max id
- Unit tests with PHPUnit (run `./vendor/bin/phpunit`)

## Run locally

```bash
cd cars-inspections-api
composer install
php -S 127.0.0.1:8000 -t public
```

Then call endpoints at `http://127.0.0.1:8000`.

Example:

```bash
curl -X POST http://127.0.0.1:8000/cars \
  -H "Content-Type: application/json" \
  -d '{"name":"Test Sedan","make":"Toyota","model":"Corolla","year":2018}'
```

## Run tests

```bash
./vendor/bin/phpunit
```
All tests are isolated unit tests and do not require a running HTTP server.

## Design Decisions & Assumptions

### Persistence Strategy

Given the 2-hour constraint, file-based JSON persistence was chosen instead of introducing a database.

To maintain scalability and separation of concerns, data access is encapsulated within Repository classes. This allows the persistence mechanism to be replaced with a relational or NoSQL database in the future without impacting controllers or business logic.

### Layered Structure

The application follows a lightweight layered structure:

- Controllers handle HTTP concerns (request parsing, response formatting, status codes).
- Services orchestrate domain logic.
- Repositories handle persistence.
- Validation classes ensure input integrity.

While the current business logic is simple, the Service layer is included to allow future extension (e.g. inspection scoring, notification triggers, maintenance workflows) without modifying controllers.

The structure aims to balance maintainability and simplicity without over-engineering.

## Inspection Domain Assumptions

Inspections are treated as a separate resource but are logically associated with cars via `carId`. Referential integrity is validated at creation time to ensure that inspections cannot be created for non-existent vehicles.

### `performedAt` Field

The `performedAt` timestamp is automatically generated by the system when an inspection is created.

Rationale:
- An inspection is assumed to be recorded at the time it is performed.
- Clients are not allowed to supply this value.
- This ensures consistency and prevents manipulation of inspection timestamps.

## Validation Decisions

### Car Year Constraints

The `year` field must satisfy:

- Must be an integer.
- Must be between 1886 and the current year.

Rationale:
- 1886 is widely recognized as the year of the first modern automobile (Karl Benz Patent-Motorwagen).
- Future production years are intentionally not allowed. Allowing future years would imply speculative or pre-production entries, which is outside the scope of this simplified API.

## Error Handling

- Validation errors return HTTP 400.
- Referencing a non-existent `carId` when creating an inspection returns HTTP 404.
- Successful resource creation returns HTTP 201 with the created resource.
- Errors are returned in structured JSON format.

For simplicity, business rule violations currently map to HTTP 404. In a production system, more granular exception types would typically be introduced to allow finer HTTP status mapping.

## Testing Scope

Due to the time constraint, testing focuses on critical paths:

- Successful car creation (happy path).
- Preventing inspection creation when the referenced car does not exist.
